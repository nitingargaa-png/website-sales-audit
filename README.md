# website-sales-audit

Skill-driven website audit + prompt-package producer for a local home-service web agency.

**Status:** Production. Actively used by the operator (Nitin) for live prospect audits and downstream handoff into `ghl-triage` and `website-audit-builder`. Three handoffs are recorded in `output/triage_handoff_log.json` against real prospects (mrrooter, mississaugaplumbingservices, warmandcoollondon) through 2026-04-20.

---

## What it does

Runs a structured 4-output website audit (`docs/SKILL.md`) on a local home-service business URL, saves the report to `output/`, and (post-audit) hands the report off to `../ghl-triage/prospect_triage.py` to score the prospect across four GHL service offerings (MCTB, VAAI, GRM, WEB). Separately, `execution/generate_website.py` assembles a three-layer prompt (universal rules + audit-derived Layer 2 + niche file) for Bolt / Lovable / Gemini / Claude Code, used to build the demo site that drives cold outreach.

The audit skill itself is executed by Claude (the human-facing skill in `docs/SKILL.md`); the Python in `execution/` is the glue that turns a saved audit into downstream artifacts (triage scores, prompt packages).

## Why it exists

(Inferred from `CLAUDE.md`, `SYSTEM_CONTEXT.md`, and `EXECUTION_GUIDE.md`.) The operator runs an AI-powered web-design agency targeting GTA-area trades — plumbing, HVAC, electrical, roofing, cleaning, etc. Stated revenue target: $10K/month. The audit doubles as a sales artifact: client-facing reports (Reports A/C) for cold outreach, and operator-only artifacts (Report B + Talking Points + TRIAGE_META block) to drive packaging and pitch. Pre-built demos generated by the same pipeline raise reply rates on cold outreach.

## Cross-repo relationships (producer side)

This repo is one of three siblings expected at `..\` relative to itself:

```
projects/
├── website-sales-audit/      ← THIS REPO — audit producer + prompt assembler
├── website-audit-builder/    ← consumer (build demos from audits)
└── ghl-triage/               ← consumer (score prospects from audits)
```

### What this repo OWNS (canonical, no parity enforcement — single source of truth)
- `docs/SKILL.md` — the audit skill itself (v12). audit-builder and ghl-triage do not have copies.
- `docs/master_prompts/universal_rules.txt` — Layer 1. Read by audit-builder at sibling-path; lives only here.
- `docs/niches/*.md` — Layer 3 (generic, plumbing, hvac, electrical, cleaning, roofing). Read by audit-builder; lives only here.
- `docs/master_prompts/*_prompt.txt` — pre-built per-niche prompts (plumbing only currently).
- `output/*.md` audit files — read upstream by both consumers.

### What this repo CO-OWNS with audit-builder (byte- or AST-identity enforced by parity hook)
Three manifests in repo root, each pinned to `website-audit-builder` via a `# sibling:` directive, are checked on every commit by `tools/check_template_parity.py` (wired into `tools/hooks/pre-commit`):

| Manifest | Granularity | Entries | Enforced by |
|---|---|---|---|
| `.shared-templates` | Whole-file byte identity (CRLF normalized) | `docs/WEBSITE_CLAUDE.md`, `tools/check_template_parity.py`, `tools/check_bom.py` | sha256 of normalized bytes |
| `.shared-functions` | Python function AST extracts, normalized | 12 functions in `execution/findings_to_layer2.py` (10 area-rule fns + `apply_structure_rules` + `rules_site_structure`) | AST `get_source_segment` + per-line `rstrip` |
| `.shared-sections` | Named text regions in `CLAUDE.md` between `<!-- shared-section:TAG start/end -->` markers | `cp1252-verification`, `commit-body-compose`, `diagram-tree-edits`, `ps5-file-handling-gotchas` | Marker-bracketed text equality |

The 12 shared Layer-2 rule functions are intentionally duplicated in both repos so each is self-contained at runtime; the parity hook catches silent drift at commit time. `generate_layer2` (the assembler) is **not** in the manifest — it is intentionally asymmetric across the two repos (this repo emits an audit-side subset; audit-builder emits the full builder Layer 2). See the comment block above `generate_layer2` in `execution/findings_to_layer2.py`.

### What this repo CONSUMES from audit-builder
`execution/generate_website.py` invokes `../website-audit-builder/execution/extract_business_data.py` via subprocess to do Firecrawl + SerpApi extraction. If audit-builder is not present as a sibling, generation fails with `FileNotFoundError`. The audit skill itself does not need audit-builder.

### What ghl-triage consumes from this repo
`execution/triage_handoff.py` calls `../ghl-triage/prospect_triage.py --from-audit <path>`. Triage parses the TRIAGE_META block at the bottom of the audit file (schema documented in `docs/WEBSITE_CLAUDE.md`) plus the Talking Points feature-detection table. The handoff log (`output/triage_handoff_log.json`) is a URL-keyed dict so re-handoffs overwrite in place rather than duplicate.

### Current TRIAGE_META schema drift state (needs verification)

Observed state on disk:

- `docs/WEBSITE_CLAUDE.md` (this repo, `main`) declares `## TRIAGE_META Schema (v1.0)` with 9 fields.
- `../website-audit-builder/docs/WEBSITE_CLAUDE.md` (`feature/v1.1-cro`) declares `## TRIAGE_META Schema (v1.1)` with 13 fields (adds `recommended_page_mode`, `page_mode_reasoning`, `estimated_monthly_leakage_usd`, `has_before_after_content`).
- `python tools/check_template_parity.py` currently exits non-zero with `MISMATCH: docs/WEBSITE_CLAUDE.md` — confirming live drift.
- The 12 Layer-2 functions and the 4 CLAUDE.md sections both pass parity.
- Both files still carry `schema_version: "1.0"` in the YAML examples, suggesting the version-table edit in audit-builder has not been fully propagated through example blocks either.

Audit-builder has advanced the schema to v1.1 on its `feature/v1.1-cro` branch; sales-audit (`main`) is still on v1.0. Note that this repo's `docs/SKILL.md` is the upstream emitter of TRIAGE_META blocks, so future schema changes will normally originate here — but the current v1.1 work is happening downstream in audit-builder. This suggests either (a) speculative consumer-side schema work pending SKILL.md updates, or (b) a workflow inversion to be resolved before propagation. **Flag as needs-verification:** confirm the intended ahead side before propagating in either direction.

## Usage

### Run an audit (this is the primary workflow)
The audit skill is invoked by Claude (Claude Code or the desktop app), not by Python. From Claude:

```
audit https://mississaugaplumbingservices.com/ and prepare full call prep
```

Claude follows `docs/SKILL.md` v12 → produces 4 narrative outputs + a TRIAGE_META YAML footer → saves to `output/[business]-[YYYY-MM-DD].md`. The file naming convention `[businessname]-[YYYY-MM-DD].md` is required because `triage_handoff.py` regex-matches it.

### Post-audit triage (auto)
Per the operator's `CLAUDE.md`, every audit run is followed by:

```powershell
python execution/triage_handoff.py --audit output/[business]-[date].md
```

Other modes:

```powershell
# Specific service only
python execution/triage_handoff.py --audit output/[file].md --service MCTB

# Batch — all untriaged audits in output/
python execution/triage_handoff.py --batch
python execution/triage_handoff.py --batch --force      # re-run already-triaged
python execution/triage_handoff.py --batch --dry-run    # preview
```

Triage results land in `../ghl-triage/output/`. The handoff log (`output/triage_handoff_log.json`) tracks state.

### Generate a demo-site prompt package

```powershell
# One shot: extract + assemble
python execution/generate_website.py --url https://example.com --mode claude-code

# Or use an existing extraction
python execution/generate_website.py --input output/structured_input.json --mode bolt

# Available modes:
#   bolt          — Bolt.new / Lovable (React + Tailwind)
#   lovable       — alias for bolt
#   gemini        — Google AI Studio (Gemini 3.1 Pro Preview)
#   claude-code   — single index.html for Claude Code in VS Code
#   skill-output  — JSON package for execute WEBSITE_GENERATION_SKILL.md
```

Output lands in `output/prompt_packages/[business]_[mode]_prompt_[YYYYMMDD].txt`.

### Batch audit
`input/urls.txt`, one URL per line (`#` comments allowed). The batch-audit loop is human-orchestrated through Claude per the operator's `CLAUDE.md` rules — there is no batch driver script for the audit phase itself.

## Key modules

### `execution/generate_website.py`
Orchestrator for the 3-layer prompt system. Loads Layer 1 (`docs/master_prompts/universal_rules.txt`), Layer 3 (`docs/niches/<niche>.md`), and Layer 2 (`findings_to_layer2.rules_site_structure(...)` injected as a SITE STRUCTURE DECISION block at prompt-assembly time). Supports five modes; the bolt/claude-code/gemini paths first try a pre-built `<niche>_<mode>_prompt.txt` and fall back to assembling from scratch. Subprocesses `../website-audit-builder/execution/extract_business_data.py` when `--url` is provided. Hard-validates extracted phone numbers (digit count, all-same-digit, supplier-length) before writing a prompt.

### `execution/findings_to_layer2.py`
The 12 area-rule functions (`rules_first_impressions` … `rules_lead_followup`) + `apply_structure_rules` + `rules_site_structure` are parity-locked to audit-builder via `.shared-functions`. They translate Claude's audit ratings (`✅` / `⚠️` / `❌`) into specific prompt instructions for the demo build. `generate_layer2`, the top-level assembler, is intentionally *not* parity-locked — see the comment above its definition; the audit-builder copy assembles the full builder Layer 2, this copy emits an audit-side subset.

### `execution/triage_handoff.py`
Post-audit subprocess driver. Resolves `TRIAGE_DIR = PROJECT_ROOT.parent / "ghl-triage"`, validates ANTHROPIC_API_KEY (falls back to reading from `ghl-triage/.env`), runs `prospect_triage.py --service ALL --from-audit <path>` with a 5-minute timeout. Maintains a URL-keyed dict at `output/triage_handoff_log.json` so re-handoffs overwrite in place. Batch mode finds files matching `*-YYYY-MM-DD.md` in `output/`.

### `tools/check_template_parity.py`
Three-manifest parity checker (templates / functions / sections). Manifests must be LF-only and BOM-free (validated at load). Each manifest pins its sibling via `# sibling: <name>` directive (env-var override). Templates: SHA-256 after CRLF normalization. Functions: AST `get_source_segment` extract, normalized whitespace, strict equality. Sections: text between `<!-- shared-section:TAG start/end -->` markers. Also enforces single-trailing-LF on every file referenced. Exit 0/1/2 (clean/content drift/infra error). Wired into `tools/hooks/pre-commit`.

### `tools/check_bom.py`
Rejects staged text files starting with a UTF-8 BOM (`EF BB BF`). Reads from the git index (`git show :path`) so working-tree changes don't influence the check. Heuristic text detection: first 8192 bytes contain no null byte. Exit 0/2/3. Exists because PowerShell `Out-File`/`Set-Content` historically default to writing BOMs.

### `docs/SKILL.md`
The audit skill itself (~155 KB, v12). Not Python — it's a Markdown skill executed by Claude. Source of truth for the audit pipeline: BANNED WORDS list, Phase 1 fetch + feature detection, the 10 audit areas, the 4 narrative output formats, the TRIAGE_META block schema, the FOLLOW-UP SEQUENCE template. Lives only here; not mirrored to sibling repos.

### `docs/WEBSITE_CLAUDE.md`
Design rules copied verbatim into every client demo-site project folder as `CLAUDE.md`. Byte-mirrored to audit-builder via `.shared-templates`. Also where the TRIAGE_META schema is documented as the contract between this repo (emitter) and audit-builder + ghl-triage (consumers).

## Configuration

Operator-tunable behaviors that aren't environment variables:

- **Niche file behavior** — `docs/niches/<trade>.md` `## PROMPT_VARIABLES` section. `generate_website.py:parse_niche_variables()` extracts `KEY: value` pairs and substitutes `[KEY]` tokens in the prompt template. Add a new niche by copying `docs/niches/_NICHE_TEMPLATE.md`.
- **Layer 1 universal rules** — `docs/master_prompts/universal_rules.txt`. Affects every site regardless of trade. Header says "Do NOT modify. Add niche rules in Layer 3."
- **Pre-built prompts** — `docs/master_prompts/<niche>_{bolt,claude_code,gemini}_prompt.txt`. If present, used in preference to layered assembly. Currently only plumbing has pre-built variants.
- **Page-mode cap** — `apply_structure_rules()` in `findings_to_layer2.py`: `cap = 6 if nav_confidence == "high" else 8`. The `PRIORITY_ORDER`, `DROP_ALWAYS`, and `PRESERVE_ALWAYS` sets there are the deterministic site-structure rules.
- **Brand color fallbacks** — `NICHE_DEFAULTS` in `rules_site_structure()`. Used when extraction doesn't pull brand colors.
- **Triage timeout** — hardcoded 300s in `triage_handoff.py:run_triage()`.
- **Parity sibling override** — set `SHARED_TEMPLATE_SIBLING`, `SHARED_FUNCTION_SIBLING`, or `SHARED_SECTION_SIBLING` env vars to redirect the parity check to a non-default sibling path.

## Known issues / gotchas

- **TRIAGE_META schema drift (needs fix / needs verification).** `python tools/check_template_parity.py` currently exits non-zero with `MISMATCH: docs/WEBSITE_CLAUDE.md`. Audit-builder (`feature/v1.1-cro`) declares `Schema (v1.1)` with 13 fields; this repo (`main`) declares `Schema (v1.0)` with 9 fields. Both copies still embed `schema_version: "1.0"` in YAML examples, indicating the version-table edit in audit-builder is incomplete. Resolution direction is open — see "Current TRIAGE_META schema drift state" under Cross-repo relationships.
- **PowerShell `python3` alias broken on this box** (per `CLAUDE.md`). Scripts use `python3` in docstrings and instructions; on the operator's Windows box, run `python` or `py` instead. `triage_handoff.py` uses `sys.executable` internally so it picks the right interpreter once invoked.
- **`docs/SKILL.md` instructions reference `python3 execution/triage_handoff.py`** (CLAUDE.md, comments throughout). Same issue — use `python` on Windows.
- **No `.env.example` in this repo.** Operator hand-creates `.env`. Sibling `website-audit-builder/.env.example` is the closest reference.
- **Hardcoded operator paths.** `CLAUDE.md` and several docs reference `C:\Users\canad\projects\website-sales-audit\` as an absolute path. Multi-machine portability is not a goal of this repo today.
- **Lovable mode is an alias for bolt.** `generate_website.py:assemble_*` doesn't have a true Lovable codepath; the suffix map collapses `lovable` → `bolt` in `load_prebuilt_prompt`.
- **`generate_layer2` is intentionally asymmetric** between this repo and audit-builder — do not enroll it in `.shared-functions`. The comment above its definition explains why; this is by design, not drift.
- **Stale `output/` test files.** `_test_applicability_*-2026-04-18.md` are leftover fixtures from applicability-gate testing; they trip the audit-file regex if you run `--batch` without `--dry-run` first. Skip pattern in `find_audit_files` only skips `triage_` / `_` prefixes — the `_test_` files start with `_` so they ARE skipped, but worth being aware of.
- **TRIAGE_META block is the contract.** Any edit to `docs/SKILL.md`'s TRIAGE_META section (Phase 2 block, around line 2478) is a breaking change requiring coordinated edits in `../website-audit-builder/execution/parse_audit.py` (v1.0 parser) and `../ghl-triage/triage/audit_parser.py`, plus a `schema_version` bump in all three places. The shared `_parse_yaml_subset` helper between audit-builder and ghl-triage is itself parity-enforced via *audit-builder's* `.shared-functions` manifest.
